# Autonomous Growth Operating System

## North Star

このサイトの最優先目的は、`skimlinks` 経由で加盟ショップへ送客し、アフィリエイト収益を最大化すること。

そのために最適化する順序は次です。

1. 検索流入を増やす
2. 流入ユーザーを、意図に合うショップ一覧や詳細ページへ最短で導く
3. ショップページから加盟ショップへ高意図で送客する
4. 送客後の成約率が高いページ構成へ寄せる
5. 収益性の高いカテゴリとショップの露出を増やす

## Confirmed Business Inputs

ユーザーから確認できている前提は次です。

- 最重要カテゴリは固定しない
- 最重要ショップも固定しない
- どのカテゴリ、どのショップを伸ばすかは `収益性` を見て都度決める
- `GSC` は見られる
- `Skimlinks` のクリック、売上、ページ別売上、ショップ別売上は見られる
- `GA` はまだ未導入で、導入したい
- 有料広告は使わない
- サイト内改善、SEO改善、商業色の強い導線、CVR最適化は許容されている
- 競合サイトの参照制約はない

このため、固定の編集方針ではなく、`実績ベースで勝ち筋を選ぶ運用` を採用する。

## Decision Rules

迷ったら、次の順で価値を判断する。

1. `収益インパクト`
   Skimlinks 加盟ショップへのクリックや成約が増えそうか
2. `検索流入インパクト`
   検索需要のあるページが増えるか、既存ページの検索意図との一致度が上がるか
3. `CVR インパクト`
   訪問者が迷わずショップへ進めるか
4. `運用継続性`
   手作業に依存せず、AI が今後も自走しやすくなるか
5. `実装コスト`
   小さな変更で大きな改善が見込めるか

## What To Prioritize

### P0

- トップページや一覧ページを、`探すサービス` として直感的に使える状態にする
- 収益性の高いカテゴリページ、ブランドページ、ショップページを厚くする
- 加盟ショップへの導線を明確にして、クリックを増やす
- 日本語検索意図に合う記事や一覧ページを増やす
- `GSC` と `Skimlinks` を使って、流入と収益の両方が強いテーマを見つける
- `GA` を導入して、ページ内行動と送客導線の改善余地を見える化する

### P1

- Skimlinks 加盟の有力ショップを優先して露出する
- ショップ詳細ページの比較情報を強化する
- ブランド軸、カテゴリ軸、用途軸でランディングページを増やす
- 内部リンクを整理して、回遊と再訪を増やす

### P2

- データ更新の自動化を進める
- テンプレート化された SEO ページ生成を進める
- CTR / CVR 改善のための文言や配置の微調整を続ける

## What Not To Prioritize

- 収益に遠い説明ページの拡充
- 見た目だけの改修
- 送客先が弱いページの量産
- 手作業前提の更新フロー

## Autonomous Loop

毎回の作業は、次のループで回す。

1. 現在のサイト構造とデータを確認する
2. `集客` `送客` `CVR` `収益性` のどこが次のボトルネックか判断する
3. 最も効果が高そうな変更を 1 つ選ぶ
4. 実装する
5. lint / 必要な確認を行う
6. commit / push まで行う
7. 次に着手すべき仮説を残す

## Current Strategic Themes

当面は次のテーマを優先する。

### 1. Search Service UX

- トップページを検索起点にする
- ショップ一覧を検索結果ページとして改善する
- カテゴリ、ブランド、人気キーワードから迷わず遷移できるようにする

### 2. Money Pages

- 収益性の高いショップ詳細ページの露出を増やす
- 実データで強いカテゴリを優先カテゴリとして昇格させる
- ブランド詳細ページから加盟ショップへ自然に送客する

### 3. Programmatic SEO

- ブランド × カテゴリ
- カテゴリ × 用途
- 日本語の検索需要がある比較・一覧意図

### 4. Measurement

- `GA` を導入して主要導線の行動計測を始める
- `GSC` で流入クエリとランディングページを確認する
- `Skimlinks` で売上寄与の高いページ、ショップ、カテゴリを確認する
- 改善対象は `流入があるのに稼げていないページ` と `稼げているのに流入が弱いページ` を優先する

## Execution Standard

通常の改善は、次まで完了してはじめて完了とする。

1. 実装
2. 確認
3. commit
4. push

ユーザーから停止指示がない限り、部分完了で止めない。

## Communication Contract

ユーザーへの毎回の細かい確認を減らし、こちらで次を決めて進める。

ただし、次のときだけ明示的に相談する。

- 収益方針に大きく影響する方向転換
- 既存の大量ページへ一括変更をかけるとき
- 外部依存や権限が必要なとき
- どの案が正しいか、事業上の判断が必要なとき

## Immediate Backlog

- `GA` を有効化して計測を始める
- トップページを検索サービスとしてさらに磨く
- ショップ一覧の検索精度と UI を改善する
- 収益性の高いカテゴリ向けのランディング導線を作る
- 加盟ショップの露出ロジックを見直す
- ブランドページから加盟ショップへの送客導線を強化する
